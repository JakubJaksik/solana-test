Kontekst: w poprzednim chacie zaprojektowaliśmy i zaimplementowaliśmy w 7 planach Rust workspace bench dla Solana fan-out tx sendingu. Bench jest niemal kompletny ale nie kompiluje się przez moją niedokończoną edycję. Plus user chce kilku zmian/uzupełnień przed pełnym smoke testem.

## 1. Co budujemy i po co

**Cel biznesowy:** HFT team na Solana DEX trader we Frankfurt'cie chce zwiększyć include ratio dla swap flow (obecnie 20-70% średnio 40-60%). Etap 1 — bench, który testuje **multi-sender fan-out** z **durable nonce dedup**: ta sama tx idzie równolegle przez N senderów (Helius, Jito, Nozomi, 0slot, bloXroute, Astralane, Syncro, Triton, AllenHark, NextBlock, BlockRazor, Harmonic), durable nonce sprawia że tylko jeden wariant ląduje (zero double-fill), zapisujemy per-sender outcome/latency do parquet.

**Pomiary z parquet → ex-post analiza:** który sender najczęściej wygrywa race do landingu, który ma najlepszą latencję, gdzie tracimy najwięcej. Bench biega open-ended aż wallet się wyzeruje.

**Kluczowe constraints HFT:**
- Frankfurt AWS host
- Najszybsza ścieżka od trigger → wysyłka → observation
- Hot path nie może mieć logów per-tx (musimy aggregat'ować)
- Dwa źródła obserwacji on-chain (SS + YS) merged on first-seen
- Nie chcemy RPC polling w hot path — local-compute / push subscriptions only

## 3. Architektura w skrócie

```
Schedule pump (deterministic chunked) ──┐
                                        ├──> Preparer (signs N variants per trigger)
NonceManager ─── take_ready() ──────────┘         │
                                                  ▼
                                              TxPool (slot,tick,sender_id)→PreSignedTx
                                                  ▲
                                                  │ take_all_for(slot,tick)
SS gRPC ──┐                                       │
          ├──> EntryMerger ──> Observer ──> Dispatcher ──> per-sender clients (HTTP/gRPC)
YS gRPC ──┘   (dedup by         (PoH tick     │              │
              entry_hash)        tracking +   │              ▼
                                 sig match)   │            SendEvent ─┐
                                              │                       ▼
                                              ▼               Matcher (AttemptState
                                       TriggerEvent           per trigger_id × sender_id)
                                                                      │
                                                                      ▼
                                                                FinalRecord
                                                                      │
                                                                      ▼
                                                              Parquet writer (61 cols)
```

## 4. Decyzje architektoniczne (już potwierdzone z userem)

1. **Durable nonce dedup** — pula 150 nonces (smoke 5), każdy trigger bierze jedno nonce → wszystkie N wariantów współdzielą `recent_blockhash = nonce_blockhash`. Pierwszy ląduje, reszta failuje validation phase za 0 lamports.
2. **Atrybucja** — per-sender tip account + 1-byte ASCII sender_id w Memo (printable range b'!'..b'~', 94 możliwości). NIE używamy różnego tip_amount per sender (psułoby aukcję).
3. **Tip strategy** — per-sender vendor minimum (Helius dual 200k, Jito 1k, Nozomi 1M, etc.) niesymetryczne, intencjonalne — mierzymy "który ląduje przy minimalnym koszcie".
4. **Cadence** — 1 tx/slot, losowy tick. Open-ended run, stop na min_balance.
5. **TxBuilder hard assert** — instruction[0] MUSI być AdvanceNonceAccount. Central builder, no instructions composed elsewhere.
6. **Per-sender protocol** — najszybszy dostępny (REST/gRPC/QUIC mix). v1 tylko REST.
7. **No in-flight cancellation** — wszystkie wysyłki dochodzą do końca naturalnie; Matcher rozwiązuje sibling status natychmiast po widzeniu pierwszego landingu (DEDUPED_TENTATIVE).
8. **Dual source observation (SS + YS)** — merged on first-seen by (slot, entry_hash). LIMITATION obecna: drugi event jest droppowany. User chce żeby OBA timestamps były zapisane w parquet (`ss_observed_at_ns`, `ys_observed_at_ns`).
9. **No RPC polling w hot path** — user explicit: SS + YS only, plus local-compute dla nonce blockhash (patrz pkt 7.5).

### 7.5 **Lokalna kalkulacja nonce blockhash z SS+YS, bez RPC pollingu**

User insight (po dyskusji): nowy nonce blockhash jest deterministyczny z chain data. Solana `AdvanceNonceAccount` ustawia nonce na:
```
new_nonce = sha256("DURABLE_NONCE" || recent_blockhash)
```
gdzie `recent_blockhash` = ostatni blockhash z BlockhashQueue banku w momencie execution (praktycznie last_entry_hash slotu S-1 dla tx wykonanej w slocie S).

SS+YS daje nam wszystkie entries per slot z ich PoH hashami. Czyli **mamy lokalnie** wszystko czego potrzeba do obliczenia nowej wartości nonce'a — bez RPC polling, bez YS accounts subscription.

**Wymagane zmiany:**
1. `observer.rs`: dodać `slot_last_entry_hash: HashMap<u64, Hash>` — cache last entry hash per slot, rolling window ~64 slotów
2. `nonce/manager.rs`: dodać metodę `on_landing_with_computed_blockhash(nonce_id, new_blockhash)` lub rozszerzyć `on_observed_landing` z parametrem
3. `matcher.rs`: gdy MatchEvent przychodzi dla naszej tx (slot S):
   - Wyciągnij `nonce_id` z AttemptRecord.reg.nonce_account_id
   - Lookup `slot_last_entry_hash[S-1]` z observer cache
   - Compute `new_blockhash = solana_sdk::nonce::state::next_durable_nonce(prev_blockhash)` (albo manual `sha256("DURABLE_NONCE" || prev_hash)`)
   - Call `manager.on_landing(nonce_id, new_blockhash)` → state → Ready { new_blockhash }
4. **Usunąć** `nonce/rpc_poll.rs` lub zostawić jako pure-fallback z bardzo długim interwałem (5-10 min) — user nie chce go w hot path
5. **Usunąć** `nonce/geyser_sub.rs` start z `run.rs` (też zbędne po lokalnej kalkulacji)

**Edge cases do uwzględnienia:**
- Slot S-1 mógł być skipped → trzeba S-2 lub mniej. Detection: jeśli `slot_last_entry_hash[S-1]` nie ma w cache, próbuj S-2.
- Race: tx ląduje w S, ale my widzieliśmy slot S-1 jeszcze nie do końca (network delay) → krótki retry w matcher z timeoutem
- Fork: jeśli slot S forkuje out, nonce się NIE zaadvansował — local-compute byłby błędny. Defense: jeśli następny `AdvanceNonceAccount` z tym computed blockhash failuje na chainie → matcher widzi error → re-compute albo fallback do RPC `getBlock(S-1).blockhash`.

**Także dla SS+YS dual observation w parquet:**
- Aktualnie merger emit'uje first-seen i drop'uje drugi event
- User chce żeby OBA timestamps były w parquet (`ss_observed_at_ns`, `ys_observed_at_ns`)
- Fix: merger emit'uje DRUGI event "second source seen" (lub osobny SecondSourceEvent channel), Matcher buduje rekord z oboma timestamps
- To **osobny fix** od nonce compute — można zrobić niezależnie

### 7.6 Hot path analysis

User chce review pod kątem HFT latency:
- **Trigger → wysyłka** path: ile czasu od observer.fire_trigger do dispatcher.send().await? Czy są zbędne hops/clones?
- **Detection** path: ile czasu od SS shred receive do matcher emit?
- **Per-thread CPU affinity** — wszystkie core_pinning options są dostępne ale nie domyślnie ustawione
- **Channel sizing**: czy 65536-element bounded queues są adequate? Czy gdzieś jest unbounded growth?
- **Lock contention**: NonceManager używa parking_lot RwLock per entry — czy fine-grained enough?
- **Memory allocations w hot path** — które rzeczy alokują w tx send loop? Można pre-allocate?
- **Tokio runtime vs std threads** — dispatcher używa tokio multi-thread; obserwer/preparer/matcher są std threads. Sensowne?
- **Pool key** `(u64, u8, u8)` — DashMap z HashSet collect przez `take_all_for` (linear scan przez wszystkie entries). Nie optimal jeśli pool jest duży. Może lepiej dwustopniowy index `Map<(slot,tick), Map<sender_id, ...>>`?

Reference: `crates/tick-trigger-bench/` ma podobną architekturę dla 1-sendera i była zoptymalizowana wcześniej (M1 trigger→send p50=0.16ms). Powinniśmy match'ować lub być szybsi.
